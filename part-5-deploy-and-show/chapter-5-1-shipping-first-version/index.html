Shipping Your First Version

Your MVP is built. It runs on your machine. Now you need to put it where real users can find it. This transition from localhost to production is where most projects die.

The gap between "it works on my machine" and "it works for users" is wider than you think. Deployment issues, environment differences, and unexpected edge cases will surface. That's normal. Here's how to cross the gap.

The Minimum Viable Deployment​

You don't need Kubernetes. You don't need CI/CD pipelines. You don't need staging environments. You need a server that runs your code and a domain pointing to it.

Here's the simplest deployment that works:

  1. Rent a $5-10 VPS from DigitalOcean, Linode, or Hetzner
  2. SSH in and set up your environment
  3. Clone your repo
  4. Run your app with a process manager like PM2 or Supervisor
  5. Set up Nginx as a reverse proxy
  6. Point your domain to the server IP
  7. Install SSL with Certbot

That's it. You can do this in an afternoon. It's not glamorous. It works.

Every solo developer I've mentored overcomplicates deployment. They research Kubernetes clusters, Docker Swarm, and multi-region failover before they have a single user. Stop. Deploy to a $5 VPS. When you hit 10,000 users, you can refactor. By then, you'll know exactly what you need.

CI/CD Basics for Solo Devs​

Continuous Integration and Continuous Deployment sound like enterprise buzzwords. For a solo dev, they're simpler: your code should go from your laptop to production with as few manual steps as possible.

A basic CI/CD pipeline for one person:

  1. Push code to GitHub (main branch)
  2. GitHub Actions or a similar tool runs your tests
  3. If tests pass, the action SSHes into your server
  4. It pulls the latest code, installs dependencies, and restarts your app

Here's a minimal GitHub Actions workflow that does this:

name: Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Deploy to VPS
        uses: appleboy/ssh-action@v0.1.5
        with:
          host: ${{ secrets.HOST }}
          username: ${{ secrets.USERNAME }}
          key: ${{ secrets.SSH_KEY }}
          script: |
            cd /var/www/myapp
            git pull
            npm install
            npm run build
            pm2 restart myapp

Set this up once. It takes an hour. After that, every git push deploys your code automatically. No more SSHing in manually. No more forgetting a step. No more "I'll deploy it later" procrastination.

You don't need unit tests covering every edge case. For your first version, a CI pipeline that just deploys on push is fine. Add tests later. The most important thing is that shipping is frictionless. If deploying requires five manual steps, you'll ship less. Period.

Domain Setup Guide​

Buy a domain. Don't use a free subdomain from a hosting provider. A custom domain costs $10-15 per year and makes your product look like a real business.

Here's the exact domain setup process:

  1. Buy your domain from Namecheap, Porkbun, or Cloudflare Registrar. Avoid GoDaddy — they upsell aggressively and make DNS management painful.

  2. Create a free Cloudflare account. Cloudflare is your DNS provider, CDN, DDoS protection, and SSL manager. All free.

  3. In Cloudflare, add your domain. It will scan existing DNS records (there won't be any yet, but it checks).

  4. Cloudflare will give you two nameservers (like dora.ns.cloudflare.com). Go to your domain registrar's dashboard and replace the default nameservers with these.

  5. Wait 5-30 minutes for propagation. Cloudflare will email you when it's active.

  6. In Cloudflare DNS settings, create these records:

    • @ A record pointing to your server IP (proxy enabled — the orange cloud)
    • www CNAME record pointing to @ (proxy enabled)
    • api A record if you have a separate API subdomain

  7. Enable "Always Use HTTPS" and "Automatic HTTPS Rewrites" in Cloudflare's SSL/TLS section.

  8. Set SSL mode to "Full (strict)" since you'll have a valid certificate on your server.

The proxy (orange cloud) is important. It hides your real server IP, caches static content, and blocks DDoS attacks. Never set DNS records to DNS-only (gray cloud) unless you have a specific reason.

SSL Configuration​

SSL is non-negotiable in 2026. Browsers mark HTTP sites as "Not Secure." Users will leave. Search engines penalize non-HTTPS sites.

Certbot is the standard tool for free SSL certificates from Let's Encrypt. Install it on your server:

sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com

Certbot automatically modifies your Nginx configuration to serve HTTPS and sets up auto-renewal. Verify renewal works:

sudo certbot renew --dry-run

If you use Cloudflare (you should), you have another option: Cloudflare Origin Certificates. These are free, never expire (well, 15 years), and are generated in the Cloudflare dashboard. This avoids the 90-day renewal cycle of Let's Encrypt.

Here's the setup:

  1. In Cloudflare SSL/TLS > Origin Server, generate a new certificate
  2. Copy the certificate and private key
  3. Save them on your server: /etc/ssl/certs/yourdomain.pem and /etc/ssl/private/yourdomain-key.pem
  4. Configure Nginx to use them

Either approach works. The point is: SSL should be configured before you share your URL with anyone.

Monitoring Tools for Solo Devs​

When your app is live, you need to know when it breaks. You can't watch a terminal all day. You need tools that watch for you.

Start with these free or cheap tools:

Uptime monitoring: UptimeRobot (free for 50 monitors) checks your site every 5 minutes and emails you if it goes down. Set it up for your main domain and any critical API endpoints.

Error tracking: Sentry has a generous free tier. Add their SDK to your app. Every unhandled exception, 500 error, and frontend crash gets reported with stack traces. You'll know exactly what broke before any user has time to email you.

Server monitoring: Netdata or htop. Netdata gives you a real-time dashboard of CPU, memory, disk, and network usage. Install it with one command: bash <(curl -Ss https://my-netdata.io/kickstart.sh). When your server slows down, you'll see why immediately.

Log management: Don't overthink this. Use journalctl for system logs and write app logs to files with timestamps. When you grow, switch to something like Logtail or Grafana Loki. For your first version, tail -f is fine.

Application Performance Monitoring: Checkly or DebugBear for frontend performance. They track page load times, Core Web Vitals, and catch regressions before users notice.

Set all of these up before launch. The cost is zero. The peace of mind is invaluable. When your app goes down at 3 AM, you'll get an email and an error trace instead of discovering it from an angry user tweet the next morning.

The Launch Checklist​

Before you share your URL with anyone, verify these:

  • Registration and login work end to end: Create a test account and run through the whole flow
  • Core flow works: The main action your product enables works without errors
  • Error states are handled: What happens when a user enters invalid data? When a network request fails?
  • The landing page makes sense: Can a first-time visitor understand what you do in 5 seconds?
  • Analytics are set up: You can see when people visit and what they do
  • You can receive feedback: An email address, a form, or a chat widget
  • SSL is working: The padlock icon shows in the browser
  • Mobile works: Open it on your phone. Does it render?
  • Database backups are automated: You can recover from a corruption or mistake
  • Environment variables are configured: No hardcoded secrets, no local-only configs
  • Rate limiting is in place: You don't want one scraper to take down your app
  • CORS is configured: Your API accepts requests from your domain only
  • Logout and session management work: Users can sign out and their session actually expires
  • Emails are sending: Password resets, welcome emails, notifications all work
  • Your domain has a custom email: hello@yourdomain.com looks professional. Use Cloudflare Email Routing (free) or Zoho (free tier).

Don't launch without checking these. Each one you skip will be the one that fails in front of your first user.

The "Ship Friday" Practice​

Here's a practice that changed how I build: Ship Friday.

Every Friday, you ship something to production. Not a huge feature. Not a perfect feature. Something. A bug fix. A small improvement. A new page. A refactor. Every single Friday.

Here's why this works:

First, it kills perfectionism. You can't spend two weeks polishing something if you need to ship in five days. You learn to ship at 80% done. 80% done is better than 100% abandoned.

Second, it builds the deployment habit. The first deployment is terrifying. The tenth is routine. The hundredth is boring. "Ship Friday" gets you to the routine stage quickly.

Third, it creates a steady stream of improvements for users. Every week, your product gets better. Users notice. They see activity and trust that you're committed.

Fourth, it prevents the "big launch" mindset. Instead of one huge launch that nobody notices, you get 52 small improvements. Some will resonate. Some won't. But you're always moving forward.

Set a recurring calendar reminder: "Ship Friday" every week at 4 PM. If you haven't shipped anything by Friday afternoon, ship the smallest thing you can. A README update counts. A changelog entry counts. The ritual matters more than the scope.

The First Deployment Always Breaks​

Accept this now. Your first deployment will have issues. Something will be misconfigured. An environment variable will be missing. A dependency will behave differently on Linux. A file path will be wrong.

This is not a sign that you're bad at deployment. It's a sign that you're doing it for the first time. Each issue you fix teaches you something about your production environment.

Document every issue you encounter. Next deployment will be smoother. The one after that will be almost boring.

Here's a concrete example of what your first deployment issues might look like:

  • Your database connection string includes localhost, but on the server, the database runs on a different host. Fix: use environment variables.
  • File upload paths use forward slashes on your Mac, but the server needs different permissions. Fix: use relative paths and proper directory setup.
  • Your app uses 500MB of memory, but your $5 VPS has 1GB and runs other services. Fix: configure swap space or upgrade to a $10 plan.
  • The email service you use blocks messages from new domains. Fix: set up SPF and DKIM records for your domain.
  • Your app crashes when it can't find the .env file. Fix: use a process manager that loads environment variables from a config file.

Document each one in a local DEPLOYMENT_NOTES.md file. When you set up your next project, you'll already know what to avoid.

The Hard Truth​

Nobody will notice your launch. You'll refresh your analytics dashboard and see zero visitors. You'll check your server logs and see only your own requests. This is normal.

Launching is not a single event. It's the start of a process. Your job now is to get your first user. Not to celebrate. Your first user will tell you more than any amount of pre-launch preparation.

The deployment itself is not the victory. It's the entry ticket. The real work starts after the green checkmark appears on your CI pipeline.

Your Turn​

Set a launch date. This week. Not next month. This week.

Go through the launch checklist above. Deploy your app. Buy your domain. Configure SSL. Set up one monitoring tool. Configure one automated backup.

Then implement Ship Friday for the next four weeks. Every Friday, ship one improvement. No matter how small.

What You Learned​

  • The simplest deployment is the best: VPS + reverse proxy + SSL
  • Set up a basic CI/CD pipeline so every push deploys automatically
  • Use Cloudflare for DNS, DDoS protection, and free SSL
  • Configure monitoring tools before you launch
  • Use a launch checklist to avoid embarrassing failures
  • Ship Friday builds the deployment habit and kills perfectionism
  • Your first deployment will break — document and fix each issue
  • Buy a custom domain and use Cloudflare for DNS
  • Nobody will notice at first — that's normal
