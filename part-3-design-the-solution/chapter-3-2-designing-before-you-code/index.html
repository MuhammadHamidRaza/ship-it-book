Designing Before You Code

You've identified the problem. You understand the system. Now you want to open your editor and start coding. Don't.

The gap between understanding a problem and building a solution is where most products die. You think you know what to build. You don't. Not until you've designed it on paper first.

The Cost of Coding Too Early​

Code is expensive to change. A wireframe takes five minutes to redraw. A database migration takes hours. A whole feature that you built and need to remove takes days.

When you code too early, every mistake costs time. When you design first, mistakes cost nothing. You catch assumptions before they become bugs. You find missing pieces before they become blockers.

I've never regretted spending too much time on design. I've never regretted building too late. I've spent months rewriting code I could have designed right the first time.

The Cost Data​

Let's talk numbers. Industry data shows that fixing a bug found in production costs 100x more than fixing the same bug found in the design phase. A requirements error caught during design costs about $100 to fix. The same error caught in production costs $10,000 or more.

This isn't because production bugs are fundamentally harder. It's because of everything wrapped around that bug. The code review. The deployment pipeline. The migration scripts. The communication with affected users. The rollback plan. The postmortem. The reputation damage.

A design problem caught at the wireframe stage is a five-minute fix. Redraw the box. Rearrange the flow. You're done. The same problem discovered after you've built the database schema, wired up the API, written the frontend components, and deployed to staging requires untangling all of that. Hours or days.

This is the fundamental economics of design-first development.

The Design Hierarchy​

Start with user flows. Before you design a single screen, map the journey. What does the user do first? What do they see next? What are the decision points? Where could they get stuck?

A user flow is a series of boxes and arrows. Each box is a step. Each arrow is a decision. Don't worry about aesthetics. Just capture the sequence.

Next, wireframes. For each step in your flow, draw the interface. Use a tool like Excalidraw, Figma, or a whiteboard. Black and white. No colors. No fonts. Just layout and content.

Wireframes test your assumptions about what information users need and what actions they take. If a wireframe feels confusing, the real interface will be impossible.

Finally, prototypes. Link your wireframes together into a clickable flow. Use a simple tool or just print them out and walk through them. Test the flow yourself. Then test it with someone else.

How to Create User Journey Maps​

A user journey map goes deeper than a flow. It captures the emotional state of the user at each step. It reveals friction points that wireframes won't show.

Draw a grid. The rows are: Stage, Actions, Thoughts, Emotions, Pain Points, Opportunities. The columns are the stages of your user's journey.

For a subscription SaaS, the stages might be: Discovery → Signup → First Use → Regular Use → Renewal.

Fill in each cell. For the Discovery stage:

  • Actions: Googles for "project management tool," reads reviews, visits your pricing page
  • Thoughts: "This looks promising but I'm not sure if it's worth the price"
  • Emotions: Curious but skeptical
  • Pain Points: Hard to compare features with competitors, no clear pricing
  • Opportunities: Add a comparison page with competitors, offer a free trial with no credit card

The emotional arc is the most revealing part. If your journey map shows a steep drop in emotion anywhere, you've found a systemic problem. A journey map for a fintech app might reveal that users feel anxious during account setup (emotional drop) and relieved when they see their first balance (emotional rise). If your journey has no emotional drop, you're not looking hard enough.

Once you have the journey map, zoom in on the pain points. These are your design priorities. Every feature you build should address a specific pain point from the map. If a feature doesn't connect to a pain point, it's optional at best and distracting at worst.

The Technical Design Document​

Before you write a single line of code, write a technical design document. Include:

  • Data model: what entities exist and how they relate
  • API surface: what endpoints you need
  • Key algorithms: complex logic explained in plain language
  • Dependencies: what external services you'll use
  • Risks: what could go wrong and how you'll handle it

This document is not for your manager. It's for you. It forces you to think through every aspect before you commit to code. If you can't describe how something works in plain language, you don't understand it well enough to code it.

Technical Design Document Template​

Here's the template I use. Copy it.

Title: [Feature Name] Technical Design

1. Goal One paragraph. What are we building and why?

2. Data Model

entities/
  User
    - id: UUID (primary key)
    - email: string (unique, indexed)
    - created_at: timestamp
  Project
    - id: UUID (primary key)
    - name: string
    - owner_id: UUID (FK to User)
    - created_at: timestamp

List every entity, every field, and the relationships. This will become your database schema.

3. API Surface

POST   /api/projects          — Create a project
GET    /api/projects          — List user's projects
GET    /api/projects/:id      — Get project details
PUT    /api/projects/:id      — Update project
DELETE /api/projects/:id      — Delete project

Define each endpoint. Include request/response shapes if they're complex.

4. Key Logic Plain language description of complex operations. No code. For example: "When a user creates a project, we need to verify they haven't reached their plan limit, create the project record, set up default permissions for the owner, and trigger a webhook to their connected Slack workspace."

5. Dependencies List every external service this feature touches. Auth. Payments. Email. File storage. Each dependency is a risk.

6. Risks and Mitigations

RiskLikelihoodImpactMitigation
Duplicate project namesLowLowAdd unique constraint on name per owner
Rate limiting from email serviceMediumMediumQueue emails, handle retries with exponential backoff

7. Open Questions List anything you're not sure about. This section is the most valuable. It surfaces assumptions before they become bugs.

When you finish the document, share it with another developer. Ask them: "What did I miss?" They will find things you didn't see. That's the whole point.

The One-Week Rule​

Spend at least one week on design for every month of expected development. If you think your MVP will take three months, spend three weeks designing it.

This sounds like a lot. It's not. The design phase uncovers problems that would take months to fix in code. The one-week rule gives you the clarity to build fast once you start coding.

Here's what happens when you skip the one-week rule. A team I worked with built a billing system for their SaaS product. They spent two days on design and six weeks on implementation. In week three of implementation, they discovered they needed to support prorated refunds. The data model didn't support it. They had to redesign the database, rewrite three API endpoints, and update all the frontend billing pages. The redesign alone took two more weeks.

Had they spent one week on design upfront, they would have listed "prorated refunds" as a key logic question in the design document. They'd have realized on day three, not week three. The one-week rule would have saved them two weeks of rework.

Case Study: A Product That Saved Months by Designing First​

A well-known story in product circles is the redesign of the Mint.com onboarding flow. The original flow took users through a 12-step setup process. Analytics showed an 80% drop-off between step 1 and step 12.

The team wanted to rebuild the entire flow. Engineering estimated three months. Instead, the product manager spent three weeks on design. She mapped the user journey. She interviewed users at each step. She discovered the real problem: users were overwhelmed by the number of choices in step 2 (categorization of expenses).

She redesigned the flow around a single insight: start with the benefit, not the setup. The new flow: "Link your bank account. Here's your spending summary. You can edit categories later." The 12 steps became 3.

Three weeks of design. Two weeks of implementation. The new flow increased completion rates from 20% to 70%.

The team saved two months of development by spending three weeks on design. They didn't build a complex solution to the wrong problem. They understood the user's emotional state and designed around it.

Tools and Workflows for Design-First Development​

You don't need expensive tools to design first. Here's what I recommend at each stage.

User flows: Paper, whiteboard, or Excalidraw. The simpler the tool, the faster you iterate. Digital tools can make you obsessed with aesthetics before you've validated the logic.

Wireframes: Balsamiq, Excalidraw, or Figma with wireframe kits. Keep it black and white. No branding. No colors. Colors create emotional reactions that distract from structural feedback. A user will say "I don't like the blue" when what they mean is "I don't understand where this button goes."

Prototypes: Figma with prototyping mode, or a simple HTML page. The goal is interactivity, not fidelity. You can even print wireframes and walk through them with a user. Low-fidelity prototypes get better feedback because users feel more comfortable criticizing them.

Technical design: A document in Notion, Google Docs, or Markdown. The format matters less than the act of writing. Writing forces precision. You can't hand-wave in a document the way you can in a conversation.

The workflow: Start with the user journey map (one session). Draw the user flows from the journey (one session). Wireframe each flow step (several sessions). Build a clickable prototype (one session). Test with someone else (one session). Write the technical design document (one session). Only then, open your editor.

Each session is 30-60 minutes. The entire process takes a week for most features. That week saves you from building the wrong thing.

The Hard Truth​

Designing first requires patience. It feels like you're not making progress. You have nothing to show for your work. No GitHub commits. No running code. Just sketches and documents.

But these sketches and documents are the most valuable output you'll create. They're your blueprint. A builder without a blueprint wastes materials. A developer without a design wastes time.

You can build a house without blueprints. It just won't be the house you want. Same with software. You can start coding without a design. You just won't ship the product you imagined.

The resistance to designing first is psychological, not practical. You feel productive when you're typing. You feel anxious when you're drawing boxes on a whiteboard. That anxiety is the feeling of learning. Sit with it. Design anyway.

Your Turn​

Take one feature you plan to build. Do not open your editor. Spend 30 minutes drawing the user flow on paper. Then spend 30 minutes sketching wireframes for each step. Then write the technical design document using the template in this chapter. Finally, create a user journey map with the emotional arc.

If you find yourself reaching for your keyboard, stop. You're not ready.

What You Learned​

  • Code is expensive to change; paper is free
  • Fixing a bug in production costs 100x more than in design
  • Design in layers: user journey map → flows → wireframes → prototypes → technical design
  • A technical design document forces you to think before you commit
  • The one-week rule: one week of design per month of development
  • User journey maps reveal emotional friction points
  • Designing first feels unproductive but saves the most time
